# PRD (Product Requirements Document) 생성

사용자의 아이디어를 구조화된 PRD로 변환합니다.

## 실행 방법

사용자에게 다음을 질문하여 정보를 수집합니다:

### 1단계: 핵심 질문
```
1. 무엇을 만들고 싶으신가요? (한 문장으로)
2. 누가 사용하나요? (대상 사용자)
3. 왜 필요한가요? (해결하려는 문제)
4. 핵심 기능 3가지는 무엇인가요?
5. 성공 기준은 무엇인가요?
```

### 2단계: 추가 질문 (필요시)
```
- 기술 스택 선호도가 있나요?
- 기존 시스템과 연동이 필요한가요?
- 마감일이나 마일스톤이 있나요?
- 참고할 유사 제품이 있나요?
```

### 3단계: PRD 문서 생성

수집한 정보를 바탕으로 아래 템플릿으로 PRD를 작성합니다:

```markdown
# [프로젝트명] PRD

## 개요
- **목표**: [한 문장 설명]
- **대상 사용자**: [누구를 위한 것인지]
- **해결 문제**: [어떤 문제를 해결하는지]

## 핵심 기능
### 1. [기능명]
- 설명: 
- 사용자 스토리: "사용자로서, 나는 ~하고 싶다, 왜냐하면 ~"
- 수락 기준:
  - [ ] 조건 1
  - [ ] 조건 2

### 2. [기능명]
...

### 3. [기능명]
...

## 기술 스택 (제안)
- Frontend: 
- Backend: 
- Database: 
- 기타: 

## 마일스톤
### Phase 1: MVP (예상: X일)
- [ ] 기능 1
- [ ] 기능 2

### Phase 2: 개선 (예상: X일)
- [ ] 기능 3
- [ ] 최적화

## 제외 범위 (Out of Scope)
- 이번 버전에서 하지 않을 것들

## 위험 요소 및 가정
- 가정: 
- 위험: 

## 성공 지표
- 지표 1: 
- 지표 2: 
```

### 4단계: 검토 및 수정

PRD 초안 작성 후:
1. 사용자에게 검토 요청
2. 피드백 반영하여 수정
3. 최종 확정 후 `docs/PRD.md`로 저장 제안

## 주의사항

- 모호한 부분은 명확히 질문
- 기술적 제약사항 현실적으로 반영
- MVP 범위는 최소화 (2주 이내 구현 가능 수준)
- 마일스톤은 작고 측정 가능하게

## 출력

PRD 문서를 `docs/PRD.md` 파일로 생성합니다.
