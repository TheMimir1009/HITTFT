# Vibe Coding Core Principles

바이브 코딩의 상세 원칙과 실천 가이드

---

## Simplicity First (단순성 우선)

### 기술 스택 선택
- AI가 잘 아는 안정된 버전 사용 (최신 버전보다 문서화 잘 된 버전)
- 사용 전 "이 라이브러리의 어느 버전을 알고 있나요?" 확인
- 복잡한 의존성보다 단순한 구조 선호

### 구현 접근법
- 가장 단순한 해결책부터 시작
- "Keep it simple" - 요청하지 않은 기능 추가 금지
- 한 파일에서 해결 가능하면 여러 파일로 분리하지 않음
- 보일러플레이트 최소화

### 프롬프트 단순화
- 하나의 프롬프트에 하나의 목표
- 여러 기능 요청 시 순차적으로 분리
- 모호한 요청은 명확화 질문 먼저

---

## Incremental Development (점진적 개발)

### 작업 분해
```
큰 기능 요청 시:
1. 전체 구조를 먼저 설명
2. 구현 단계를 3-5개로 분해하여 제안
3. 각 단계별 승인 후 진행
4. 단계 완료마다 테스트 가능한 상태 확인
```

### 테스트 가능한 단위
- 각 변경은 독립적으로 테스트 가능해야 함
- "이 변경 후 어떻게 테스트하면 될까요?" 안내 제공
- 에러 발생 시 마지막 작동 상태로 롤백 용이하게

### 리팩토링 시점
- 기능이 작동한 후에 리팩토링
- 리팩토링과 기능 추가를 동시에 하지 않음
- 중복 코드 발견 시 별도 단계로 정리 제안

---

## Context Management (컨텍스트 관리)

### 효과적인 정보 제공
- 관련 파일만 언급 (불필요한 파일 로드 방지)
- 에러 메시지는 전체 복사보다 핵심 부분 강조
- 스크린샷/시각 자료 활용 권장

### 컨텍스트 윈도우 관리
- 긴 대화 후에는 새 채팅 시작 권장
- 이전 컨텍스트가 필요하면 요약하여 전달
- 관련 없는 과거 대화 내용은 언급하지 않음

### 세션 간 연속성
- 중요한 결정사항은 주석이나 문서로 기록 제안
- 작업 중단 시 현재 상태와 다음 단계 명시
- plan.md 또는 TODO 파일 활용 권장

---

## Problem Solving (문제 해결)

### 에러 대응 순서
```
1. 에러 메시지 정확히 읽고 이해
2. 에러 위치와 원인 파악
3. 최소한의 변경으로 해결 시도
4. 같은 접근 3회 실패 시 다른 방법 제안
5. 해결 불가 시 원인 설명 + 대안 제시
```

### 디버깅 원칙
- 한 번에 하나의 변수만 변경
- 변경 전후 상태 비교 가능하게
- 임시 해결책(hack)은 주석으로 표시

### 막혔을 때
- 같은 프롬프트 반복 대신 다르게 재구성
- 문제를 더 작은 단위로 분해
- "지금까지 시도한 것" 정리 후 새 접근

---

## Code Generation Guidelines (코드 생성 가이드)

### 생성 전 확인
- 기존 코드 패턴 파악
- 프로젝트의 네이밍 컨벤션 따르기
- 사용 중인 라이브러리 버전 확인

### 생성 코드 품질
- 자명하지 않은 로직에 주석 추가
- 매직 넘버 대신 상수 사용
- 에러 처리 포함

### 생성 후 설명
- 핵심 로직 간단히 설명
- 수정이 필요할 수 있는 부분 명시
- 테스트 방법 안내

---

## Anti-Patterns (피해야 할 패턴)

### 하지 말아야 할 것
- ❌ "전체 앱을 만들어줘" 시도
- ❌ 에러 무시하고 다음 기능으로 진행
- ❌ 이해 못한 코드 그대로 사용
- ❌ 테스트 없이 여러 파일 동시 수정
- ❌ 작동하는 코드를 불필요하게 리팩토링

### 위험 신호
- 같은 에러가 3회 이상 반복 → 접근 방식 재검토
- 변경 범위가 계속 확대 → 스코프 다시 정의
- "거의 다 됐어요" 반복 → 체크포인트로 롤백 고려
